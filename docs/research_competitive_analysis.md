## 一、主流 BI 平台交互模式

### 1.1 查询构建 3 种范式

| 范式 | 代表 | 核心机制 | 本项目对标 |
|------|------|---------|-----------|
| **笔记本卡片式** | Metabase | 每个 SQL 子句 = 一张可视化卡片（筛选→聚合→JOIN→排序），可拖拽排序 | 向导模式 5 步接近此模式，但缺少卡片交互 |
| **SQL 编辑器 + 可视化增强** | Redash | 大文本区 + Schema 浏览器 + `{{param}}` 参数化 + 查询片段复用 | 本项目 SQL 导入+解析 已优于 Redash |
| **语义层 + 双模式** | Superset | 数据集预定义指标/维度 → 图表构建器选择，也可切 SQL Lab | 接近本项目的"向导+RawSql"双模式 |

**推荐采纳：** Metabase 的笔记本卡片模式（每步一张独立编辑卡片）+ Superset 的语义层概念（预定义指标供图表构建器选用）。

### 1.2 筛选交互模式

- **侧边栏筛选面板**（Superset）：独立区域，类型感知控件（日期→日历、类别→下拉、数值→滑块），修改即自动重查（300ms 防抖）
- **级联筛选**（Grafana）：模板变量链式依赖，选省份→城市下拉自动刷新
- **跨图表联动**（Metabase/Superset）：点击图表数据点 → 过滤仪表盘所有卡片

### 1.3 JOIN 配置 UI

仅 Metabase 提供可视化 JOIN 构建器：下拉选关联表（FK 自动推荐）、选 JOIN 类型（Venn 图图标）、映射源/目标列、"+"按钮追加条件。其他平台或写入 SQL，或在数据集定义阶段处理。

### 1.4 聚合/分组控制

Metabase "Summarize" 侧边栏：选择指标函数（Count/Sum/Avg）+ 分组维度（多列拖拽排序），实时预览。支持多聚合叠加（如 Count + Sum of Revenue）。

### 1.5 图表选择

四平台统一为**图标网格**展示：Metabase ~15 种、Superset ~60+ 种。关键模式：
- 上下文验证（灰色不可用的图表类型）
- "必需字段"提示
- 右侧配置面板（轴、颜色、图例、标签）

### 1.6 仪表盘布局

| 平台 | 网格 | 交互 |
|------|------|------|
| Metabase | 12 列固定网格 | 拖拽排列 + 右下角调整大小 |
| Superset | CSS Grid (React Grid Layout) | 自由拖拽 + Tab 分组 |
| Grafana | 24 列 + 行折叠 | 自由拖拽 + 面板叠加 |

### 1.7 下钻（Drill-down）4 种模式

| 模式 | 代表 | 机制 |
|------|------|------|
| **点击过滤** | Metabase | 点击柱状图 → 弹出菜单"查看这些数据"/"按相关维度拆分" |
| **数据链接** | Grafana | URL 模板 + 变量替换 → 跳转另一仪表盘 |
| **浏览起点** | Grafana/Superset | 每张卡片标题栏"Explore"按钮，打开预填充查询编辑器 |
| **声明式下钻** | Superset | 图表配置中定义下钻维度和顺序（时间→地区→城市→门店） |

### 1.8 端到端工作流

```
数据连接 → 数据准备（表/视图/预 JOIN） → 查询探索（可视化或 SQL）
→ 选图表类型 → 配置映射（X/Y/系列/颜色） → 保存
→ 仪表盘组合（拖入卡片 + 全局筛选 + 跨图表联动） → 发布分享
```

## 二、开源 HIS/医院 BI 平台功能

### 2.1 OpenEMR

- **报告模块**：临床质量报告（CQM）、收入分析、预约报告、处方报告
- **思路**：引入"报告模板"概念——查询配置 + 标题页/页眉/页脚 + 导出格式选择
- **数据范围**：用户权限限定于特定诊所/科室，可参考实现"数据范围过滤器"

### 2.2 OpenMRS

- **报告框架**："报告定义" → "数据集定义" → "指标"三层抽象
- **可组合指标**：基础查询 + 嵌套指标（条件计数/去重计数/百分比）
- **多查询报告**：多个数据集编排为多页报告，类似本项目的"单查询"→"多查询报告"升级

### 2.3 Odoo 医院管理

- **闭环分析**：医疗数据 + 财务 + 库存 + 人力资源联动
- **计算字段**：用户自定义表达式列（如 `SUM(cost)/COUNT(*)` 作为均次费用）

### 2.4 中国 HIS 厂商通用功能

- **门诊统计**：日均门诊量、高峰时段、医生平均看诊量、复诊率
- **住院统计**：床位周转率、平均住院日（ALOS）、术前平均住院日、DRG 分组
- **收入分析**：总收入、按科室/收费类型/支付方式细分、医保 vs 自费、欠费分析
- **药品统计**：药品收入占比、抗菌药物使用率、基药使用比例、处方点评
- **DRG/DIP 分析**：RW 权重、CMI 病例组合指数、盈亏分析、低/高倍率病例
- **手术统计**：手术量（按级别/科室）、麻醉方式、术后并发症率

### 3.2 8 大统计类别及核心指标

#### 门诊统计
日均门诊量、初诊 vs 复诊比例（FIRST_VISIT_INDICATOR）、科室分布（VISIT_DEPT）、号别分布（普通/专家，CLINIC_TYPE）、医生工作量、挂号费收入、退号率、患者身份/费别构成、高峰时段分析、门诊诊断分布

#### 住院统计
入出院人数、在院人数（期末）、床位使用率（占用床日/开放床日）、平均住院日（AVG(DISCHARGE_DATE - ADMISSION_DATE)）、床位周转次数、入出院诊断符合率、治疗结果分布（治愈/好转/未愈/死亡）、危重患者比例、抢救成功率、术前平均住院日、转科率、3/7/30日再入院率

#### 医技统计
检查人次（按类别）、检验工作量（按科室/类别）、检查阳性率、检查收入、设备使用率、平均等待时间

#### 手术统计
手术总台数、手术科室分布、大/中/小手术构成（OPERATION_SCALE）、麻醉方法分布、切口愈合情况（WOUND_GRADE + HEAL）、手术并发症发生率、手术收入、手术室利用率、术前访视率

#### 收入统计
门诊/住院总收入、收入构成（药品/检查/治疗/材料/护理/其他，按 ITEM_CLASS）、次均门诊费用、次均住院费用、日均住院费用、医保 vs 自费、开单科室 vs 执行科室收入、预交金余额、欠费情况、支付方式构成

#### 疾病统计
出院患者疾病构成（ICD 编码）、单病种统计（特定 ICD 范围的患者数/费用/住院日）、死因构成、年龄别疾病分布、季节性疾病分布、疾病分类前 10 位

#### 医疗质量
入出院诊断符合率、手术前后诊断符合率、七日确诊率、医院感染率、病死率、尸检率、医疗事故/差错、病案质量（MR_QUALITY）

#### 工作效率
医师人均日诊疗人次、医师人均管床数、护理人员人均管床数、床位周转间隔、平均床位工作日

### 3.3 卫健委标准报表（9 张法定表）

1. **卫统 1 表** — 医疗卫生机构基本情况
2. **卫统 2 表** — 人力资源情况
3. **卫统 3 表** — 床位设备情况
4. **卫统 4 表** — 医疗服务量（门急诊人次、出院人数）
5. **卫统 5 表** — 医院工作效率（床位使用率、平均住院日）
6. **卫统 6 表** — 医疗收入与费用
7. **卫统 7 表** — 住院病人疾病分类（ICD-10）
8. **卫统 8 表** — 手术及操作分类
9. **卫统 9 表** — 医疗质量安全

三级医院等级评审另有 50+ 项核心指标（CMI 值、低风险死亡率、抗菌药物使用率等）。

### 3.4 报表层次结构

```
医院统计
├── 日报：门诊工作日报、病房工作日报、急诊工作日报、医技工作日报
├── 月报：门诊/住院/医技/手术/收入/质量/效率/疾病 8 张月报
├── 季报/半年报：医疗指标季报、经营分析季报
├── 年报：年度统计汇编、单病种年报、疾病分类年报、医疗质量综合评价
└── 专题分析：同比/环比对比、趋势分析、目标完成分析
```

## 五、建议优先落地功能

### P0 — 数据范围权限（上下文过滤器）

**为什么：** 医生只能看自己科室数据、科主任看全科、院长看全院。多用户场景的前提条件。

**怎么做：** 利用已有 `QueryFilter` + JWT Claims 自动注入 `CurrentDeptId`/`CurrentUserId`。在 `QueryFilter` 加 `IsContextFilter` 和 `ContextKey`，`QueryExecutionService` 执行时自动注入，用户不可覆盖。

### P1 — 交叉表/数据透视表

**为什么：** 中国医院统计"科室 × 月份"交叉表是最高频需求。门诊量月报、收入构成分析都依赖此格式。

**怎么做：** `QueryConfig` 加 `PivotRowField`/`PivotColumnField`/`PivotValueField`/`PivotAggregateType`。后端用条件聚合 `SUM(CASE WHEN month=1 THEN value END)` 生成 SQL。前端用 ECharts 或自定义 HTML 表格渲染。

### P1 — 下钻联动

**为什么：** 仪表盘卡片点击 → 明细数据，是 BI 体验的核心闭环。

**怎么做：** `DashboardCard` 加 `DrillDownQueryConfigId`。ECharts `click` 事件捕获数据点 → 带筛选参数跳转到 `QueryView`。

### P2 — 审计日志

**为什么：** 三级医院评审合规要求。追踪谁查询了什么、何时、结果。

**怎么做：** 新建 `QueryLog` 表（`UserId`, `QueryConfigId`, `Timestamp`, `Filters`, `RowCount`, `ElapsedMs`）。每次 `ExecuteAsync` 执行后写入，异步不阻塞查询。

### P2 — 预构建统计模板包

**为什么：** 医院标准统计指标是规范的，提供开箱即用的模板可大幅加速上线。

**怎么做：** 创建 `seed_data/` 目录，将门诊量日报、住院月报、收入分析等标准查询导出为 JSON/YAML。前端提供"导入标准模板"按钮。

### P2 — 定时报告/订阅

**为什么：** 医院管理者需要日报/周报/月报自动推送，而非每次手动查询。

**怎么做：** `QueryConfig` 加 `ScheduleCron`/`ScheduleEnabled`。后台 `IHostedService` 定时执行预缓存查询并生成 Excel 文件或邮件。

### P3 — 图表类型扩展

在现有 4 种图表（数字/柱/线/饼）基础上增加：堆叠柱状图、雷达图、散点图、热力图、仪表盘。

### P3 — 同比/环比计算

**为什么：** 几乎所有医院报告都要求与去年同期/上月对比。

**怎么做：** `QueryConfig` 加 `CompareType`（YoY/MoM）。执行时自动生成同期查询并计算差值/增长率。

## 六、关键编码与数据注意事项

1. **优先使用预统计中间表**：第 19 节的 `QU_*` 表比原始事务表查询效率高得多，应优先用于统计报表
2. **三层时间维度**：日数据（`DEPT_*_DAY`）、月数据（`QU_*`）、年数据（月汇总或年度 WHERE）
3. **Oracle 10g 分页**：必须用 `ROWNUM` 三层嵌套子查询，不支持 `OFFSET/FETCH`
4. **US7ASCII 编码**：现有 `ConvertEncoding` 逻辑需在统计查询中保留
5. **费用直接取自事务表**：`INP_BILL_DETAIL.CHARGES` 已为实际收费金额，无需再计算价格系数
6. **科室代码用层次码**：统计报告用最小统计单位的临床科室
7. **日期参数化**：统一使用 `TO_DATE(:param, 'YYYY-MM-DD')` 模式
